# AR Word Chain 実装設計書 v2

**Team Hero** — Mixed Reality 課題
**対象実装**: Unity (AR Foundation) + Python (Object Detection Server)

> **v2 改訂ポイント**: 「カメラに映ったら自動判定」から「**候補表示 → ユーザのタッチで確定**」というインタラクションモデルに変更。HCI 要素（ジェスチャ、フィードバック、操作感）を中心に再設計。

---

## 1. プロジェクト概要

### 1.1 目的
スマートフォンのカメラで現実世界を撮影し、検出された物体名を使って **English Word Chain（しりとり）** を行う AR ゲームを実装する。
**ユーザは候補として表示された物体をタッチして単語を「宣言」することで、能動的にしりとりを進める。**

### 1.2 なぜタッチ確定型か（HCI 的根拠）

| 自動判定モデル                                | タッチ確定モデル（採用）                            |
|-----------------------------------------------|-----------------------------------------------------|
| カメラを向けただけで判定が走る                 | ユーザが「これだ」と選んで宣言する                  |
| 受動的・運任せ                                 | 能動的・戦略的（複数候補から選ぶ）                  |
| 誤検出が即失点になる                           | 誤検出はタッチしなければ無視できる                  |
| 「探した感」「見つけた感」が弱い               | 探索 → 照準 → 確定 の操作で達成感が出る             |
| AR らしさが薄い                                | 実物体をタップする身体性が体験に直結               |

タッチによる **コミットメント（commitment）** がゲームに「決断」を持ち込み、しりとりの本質である語彙選択を主体的にする。

### 1.3 インタラクションモデル
```
   [ Explore ] ──カメラを動かす──▶ [ Aim ] ──タップ──▶ [ Commit ]
       ▲                              │                    │
       │                              │ 候補を眺める       │ 判定 & エフェクト
       └──────── 失敗 ────────────────┘                    │
                                                          ▼
                                                   [ Next Turn ]
```

## 2. ゲームルール（改訂）

### 2.1 基本ルール
- 画面には検出された物体に **半透明の候補バウンディングボックス + ラベル** がリアルタイムで重畳表示される（**未確定状態**）。
- ユーザは候補の中から **タップで一つ選び宣言** する。
- 宣言された単語の **先頭文字** が、直前単語の末尾文字と一致すれば **Good!**。
- 不一致・既出ならば **Bad!**（小ペナルティ or タイマー消費）。

### 2.2 候補の表示ルール（HCI 設計）
- **マッチする候補**（先頭文字 OK & 未使用）: **緑の枠 + ハイライト**で「タップ可能」感を強調
- **マッチしない候補**: **グレーの枠**で「見えてるけど無効」を表現
- **既出単語**: **斜線入りグレー**で「もう使った」を表現
- **信頼度低 (<0.5)**: 表示しない or 点滅で曖昧さを伝達

これにより、ユーザは「**どれが正解候補か**」を視覚的に即時把握できる（**アフォーダンスの可視化**）。

### 2.3 ジェスチャ仕様

| ジェスチャ              | 動作                                                       | HCI 的意図                                |
|-------------------------|------------------------------------------------------------|-------------------------------------------|
| **シングルタップ**      | bbox 内をタップ → 単語を宣言                                | 主要なコミット操作                        |
| **ロングプレス (0.5s)** | 物体の詳細情報（候補単語の他案、信頼度）をポップアップ      | 情報探索（曖昧時のヘルプ）                |
| **ダブルタップ**        | スキップ宣言（行き詰まり時）                                | 救済操作                                  |
| **シェイク**            | ヒント表示（次に必要な文字で始まる物体例を提案）            | スタック状態の打開                        |
| **ピンチアウト**        | 該当物体にズーム（カメラズーム or AR ラベル拡大）           | 細部確認                                  |

MVP では **シングルタップ + ダブルタップ + シェイク** の 3 つを必須実装、他は任意。

## 5. データフロー（1 ターン・改訂）

```
1. 探索フェーズ:
   - ユーザがカメラを動かす
   - Unity が 5 fps で JPEG をサーバへ送信
   - サーバから detections + tracking_id を受信
   - CandidateManager が状態タグ付け
   - BBoxRenderer が色分けされた枠を画面に重畳

2. 照準フェーズ:
   - ユーザは緑枠の候補を視認
   - 必要なら長押しで詳細確認、シェイクでヒント取得

3. 確定フェーズ:
   - ユーザが緑枠（または任意の枠）をタップ
   - InputDispatcher が画面座標 → 最前面 bbox を解決
       → 重なりは「面積最小 (= 最も精密に指したもの)」を優先
   - CandidateManager.GetCandidate(trackingId) → Detection 取得
   - WordChainGameManager.Commit(detection) を呼ぶ

4. 判定フェーズ:
   - Commit 内部:
       a. word = detection.label.lower()
       b. word[0] == required_letter ?
       c. word in used_words ?
       d. → JudgeResult { Good / Bad / AlreadyUsed }
   - 結果に応じてエフェクト + サウンド

5. アンカー固定:
   - 成功時のみ、bbox 中心から ARRaycast → 平面 Hit 位置に Anchor 配置
   - ラベル "Apple ✓" を 3D テキストで永続表示（ゲーム終了まで）

6. 状態更新:
   - required_letter = word[-1]
   - used_words.Add(word)
   - スコア +10、UI 更新

7. 次の探索フェーズへ
```

#### `WordChainGameManager.cs` (改訂)
```csharp
public JudgeResult Commit(Candidate c) {
    var word = c.label.ToLower();
    if (usedWords.Contains(word)) return JudgeResult.AlreadyUsed;
    if (word[0] != requiredLetter)  return JudgeResult.Bad;

    usedWords.Add(word);
    score += 10;
    requiredLetter = word[word.Length - 1];
    OnCommitted?.Invoke(c, JudgeResult.Good);
    return JudgeResult.Good;
}

public void Skip() {
    // ダブルタップでスキップ。ペナルティ -5、タイマー -3s 等
    requiredLetter = NextLetterInAlphabet(requiredLetter);
    score -= 5;
}

public List<string> Hint() {
    // 使用可能語彙から requiredLetter で始まる3つを返す
}
```

#### `AnchorPlacementService.cs` (改訂)
- 確定時のみ呼ばれる
- bbox 中心 → screen-space → ARRaycast(plane)
- Hit すれば Anchor、しなければカメラ前方 1m にビルボード Fallback

### 6.2 タッチ → bbox 解決アルゴリズム

複数の候補が重なる場合、以下の優先順位で 1 つを選ぶ：

```
1. screenPos が bbox 内に含まれる候補をフィルタ
2. その中で「面積最小」のものを優先（小さい物体を精密に指したと解釈）
3. 同面積なら confidence が最大のもの
4. それでも複数なら最前面（手前の AR Anchor）を選ぶ
```

これにより、机の上のリンゴ（大きい bbox）と、その上のフォーク（小さい bbox）が重なっていても、フォークを意図的に選べる。

### 6.3 視覚フィードバック設計

| イベント       | 視覚                                       | 聴覚                  | 触覚 (Haptic)         |
|----------------|--------------------------------------------|-----------------------|-----------------------|
| 候補出現       | bbox がふわっとフェードイン                | （無音）              | なし                  |
| Match 候補     | 緑枠 + ゆっくり脈動                        | （無音）              | なし                  |
| ホバー（任意）  | bbox がわずかに拡大                        | （無音）              | なし                  |
| Tap (Good)     | 緑のパーティクル爆発 + "✓" 大型表示        | 上昇音 + 拍手         | 短い振動 (success)    |
| Tap (Bad)      | 赤の "✗" + 画面振動                        | 下降音                | 二回振動 (warning)    |
| Tap (Used)     | 黄色の "Already used" トースト             | 短いブザー音          | 短い振動              |
| Skip           | 画面外へ単語がスライドアウト                | スワイプ音            | 短い振動              |
| Hint           | ヒント候補の例がカード状にスライドイン      | キラキラ音            | なし                  |

**設計原則**: フィードバックは **即座 (<100ms) ・多感覚 (視+聴+触) ・bbox 位置で起こす**。グローバルな画面エフェクトだけだと「何にタップしたか」の対応が崩れる。

## 7. Python 側 詳細設計（改訂）

### 7.1 追跡 ID の付与

`ultralytics` の組み込み tracker（ByteTrack / BoT-SORT）を使う：

```python
results = self.model.track(
    img_bgr, conf=0.5, iou=0.45, persist=True, tracker="bytetrack.yaml"
)
# results[0].boxes.id にトラッキング ID が入る
```

これにより、同じ物体には連続フレームで同一 ID が振られる。Unity 側で bbox がちらつかず、ユーザがタップ照準しやすくなる。

### 7.2 API レスポンス（拡張）

```json
{
  "frame_id": 123,
  "elapsed_ms": 42,
  "image_size": {"w": 1280, "h": 720},
  "detections": [
    {
      "tracking_id": 7,
      "label": "apple",
      "confidence": 0.92,
      "bbox": {"x": 0.42, "y": 0.31, "w": 0.18, "h": 0.22}
    }
  ]
}
```

`tracking_id` が追加された以外は v1 と同じ。

## 10. 想定リスクと対策（改訂）

| リスク                                              | 対策                                                       |
|-----------------------------------------------------|------------------------------------------------------------|
| bbox がちらついてタップしづらい                     | tracking_id ベースで Unity 側に EMA スムージング適用       |
| 重なる物体のタップ判定が曖昧                        | 面積最小優先のヒットテスト + 軽いハプティクスで確定感を補助|
| 候補が多すぎて画面が情報過多                        | Match 候補のみ大きく、その他は枠を細く・薄くする            |
| ダブルタップが意図せず発動                          | 閾値を 0.3s 以内 + bbox 外でのみ受付など条件付け            |
| シェイクが歩行中に誤発動                            | しきい値を高めに、確認ダイアログを挟む                      |
| Match 候補が一つも視野にないと詰む                  | 一定時間後に自動でヒント表示、または「視野を広げて」と促す  |
| AR Anchor が増えすぎてシーンが重い                  | 直近 N 個のみ保持、古いものはフェードアウト                 |

## 11. 評価観点（HCI 視点での発表用ネタ）

授業発表で「HCI 的にちゃんと設計した」と主張するための観点：

1. **アフォーダンス**: 緑の脈動枠が「タップ可能」を視覚的に伝える
2. **フィードバックの即時性・多感覚性**: 視+聴+触で確定感を出す
3. **エラー寛容性**: 誤検出はタップしなければ無視できる
4. **救済機構**: スキップ・ヒントで詰まりを回避
5. **学習可能性**: 色のセマンティクス（緑=可、灰=不可、斜線=済）が直感的
6. **身体性**: 実物体をタップする行為が AR の没入感を強化

簡単なユーザテスト（チームメンバー間で 5 分プレイ × 4 人）を行い、上記観点で気づきをまとめれば発表のオチに使えます。

## 13. v1 → v2 サマリ

| 項目             | v1                          | v2                                              |
|------------------|-----------------------------|-------------------------------------------------|
| 判定トリガ        | カメラに映ったら自動         | **タッチで確定**                                |
| 候補の扱い        | 検出即判定                  | **未確定プール** → 状態色分け → タップで確定    |
| 追跡 ID          | なし                        | **ByteTrack で付与**（ちらつき防止）            |
| Anchor 配置       | 検出時                      | **確定時のみ**                                  |
| ジェスチャ        | なし                        | Tap / DoubleTap / LongPress / Shake             |
| フィードバック    | 視覚のみ                    | **視 + 聴 + 触**                                |
| HCI 的主張点      | 弱                          | アフォーダンス / 救済機構 / 身体性 を明示       |
